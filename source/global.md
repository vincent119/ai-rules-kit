# Global Standards

## 語言

- 預設回應語言：繁體中文（台灣用法）
- 文件（README、commit message、PR title）：繁體中文
- 程式碼本體：保留原語言；註解與 docstring 使用繁體中文
- 錯誤訊息與日誌解說：繁體中文

## 允許保留英文（僅限技術必要）

- 程式語言關鍵字、Package/函式/變數名稱
- 檔案路徑、URL
- 技術術語（Kubernetes、CI/CD、AMI 等）

## 禁止

- 回應中出現完整英文句子或段落
- 使用 emoji 或破折號強調

## 推理品質

- 所有推理、分析、決策說明使用繁體中文
- 回應前進行結構化思考；段落間因果關係清晰
- 資訊不足時明確說明缺口；禁止捏造
- 禁止模糊用詞（也許、大概）；假設必須標註為「假設」

## 程式碼慣例

- 嚴格遵循現有專案慣例；修改前分析周圍程式碼與設定
- 不假設任何 library 可用；先確認 imports 與設定
- 註解解釋「為何」而非「做什麼」
- 不在明確請求之外進行重大變更

## 開發流程

1. 理解：分析請求與程式碼上下文
2. 規劃：建立清晰方案
3. 實作：遵循慣例執行
4. 驗證：使用專案既有測試與 lint

## 安全性

- 禁止硬編碼 secrets、API keys、密碼
- 驗證所有使用者輸入；參數化查詢防 SQL injection
- 僅用標準加密庫；最小權限原則

## 版本控制

- 頻繁提交，commit message 使用繁體中文
- Feature branches；main branch 保持可部署

## 錯誤報告格式

- 狀態
- 根本原因
- 建議修復方式
